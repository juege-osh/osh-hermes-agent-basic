# 工具系统与终端后端

## 为什么这是 Hermes 最有价值的一层

因为只有“会调用工具”，Hermes 才从一个聊天机器人变成一个真正能干活的 agent。

## Tool 和 Toolset 是什么

### Tool

单个工具能力，比如：

- 执行终端命令
- 搜索网页
- 读写文件

### Toolset

一组按主题打包的工具集合。

## 小白一定要先理解的事

工具不是越多越好。

工具越多，通常意味着：

- 权限更大
- 复杂度更高
- 出错面更广
- 安全边界更难控

所以新手建议从“最小可用工具集”开始。

## 工具的真实风险

很多人只看到“能自动执行命令”，没看到另外半句话：

“也就意味着它可能执行错命令。”

因此你需要建立 3 个意识：

1. 可执行不等于该执行
2. 本地执行不等于安全
3. 自动化程度越高，越需要权限设计

## Hermes 的终端 backend 是什么

Hermes 支持多种终端执行后端。根据官方文档，常见包括：

- `local`
- `docker`
- `ssh`
- `singularity`
- `modal`
- `daytona`

## 用人话分别解释

### local

就在当前机器上执行命令。

优点：

- 简单
- 快
- 最适合本地学习

风险：

- 直接碰你本机环境

### docker

在容器里执行命令。

优点：

- 隔离更强
- 更适合长期运行

缺点：

- 配置更复杂

### ssh

通过 SSH 到另一台机器执行。

适合：

- 远端服务器运维
- 远端部署任务

### modal / daytona / singularity

更偏云端或特定运行环境，适合后期扩展，不建议新手第一天就上。

## 新手建议的 backend 选择

### 第一阶段

用 `local`

### 第二阶段

需要隔离时再考虑 `docker`

### 第三阶段

需要远端执行时再考虑 `ssh`

## 实操建议

### 先查看工具配置

```bash
hermes tools
```

### 再看配置项

```bash
hermes config set terminal.backend local
```

## 什么时候该换成 Docker backend

以下任意情况出现，就值得考虑：

- 你不想让 agent 直接碰本机系统
- 你准备长期跑自动化任务
- 你要把环境搬到服务器
- 你希望不同项目各自隔离依赖

## 容器化为什么不是“高级玩家专属”

不是因为酷，而是因为更稳。

尤其当你让 agent：

- 安装依赖
- 运行脚本
- 修改文件
- 长时间执行任务

隔离环境能大幅减少“把宿主机搞脏”的风险。

## 什么时候不要急着上复杂工具

如果你现在还没跑通这些最基础动作：

- 正常对话
- 正常换 model
- 正常保存会话

那就先不要上复杂工具体系。

## 关于 sudo

终端工具最容易带来权限问题。你要清楚：

- agent 能执行 sudo，不等于你应该默认给它 sudo
- 长期服务更不建议宽松给权

## 学习顺序建议

1. 先理解工具是“执行能力”
2. 再理解 backend 是“执行环境”
3. 最后再理解权限与隔离

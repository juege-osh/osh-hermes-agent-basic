# Memory、Skills、MCP 到底有什么区别

## 为什么很多人会搞混

因为它们看起来都像在“给 Hermes 加能力”，但三者的作用层级完全不同。

## 一句话区分

- `Memory`：让 Hermes 更记得你和过去发生过什么
- `Skill`：让 Hermes 遇到某类任务时有现成做法
- `MCP`：让 Hermes 接入外部系统提供的新工具或上下文

## Memory 是什么

Memory 更像长期记忆层。

它解决的问题是：

- 这个用户常用什么
- 过去完成过哪些事情
- 哪些偏好值得长期保留

## Session 和 Memory 不是一回事

### Session

偏当前会话历史。

### Memory

偏跨会话、跨时间的持续信息。

## Skill 是什么

官方文档里，Skill 本质上是一份会按需加载的任务说明书。

它通常包含：

- 名称
- 描述
- 什么时候该用
- 具体步骤

## 小白怎么理解 Skill

你可以把它理解成：

“给 agent 的标准作业流程卡片。”

比如：

- 部署到某平台
- 创建 PR
- 调用某外部服务

## MCP 是什么

MCP 是 Model Context Protocol。

你现在不需要死背协议细节，只要先知道：

- 它可以把 Hermes 连接到外部能力源
- 这些能力可能是工具、数据、上下文

## 用场景来理解三者

### 场景 1：你经常写同一种部署任务

更适合 Skill。

### 场景 2：Hermes 需要记住你的偏好

更适合 Memory。

### 场景 3：Hermes 需要访问 GitHub、数据库或某个外部系统能力

更适合 MCP。

## 新手最容易踩的坑

### 坑 1：以为 Skill 会自动永久记住你的一切

不会。Skill 不是长期记忆。

### 坑 2：以为 Memory 能替代流程设计

不会。Memory 记住偏好，不等于帮你定义标准流程。

### 坑 3：以为接了 MCP 就一定更稳定

不一定。MCP 增加了能力，也增加了依赖与故障点。

## 学习建议

### 第一阶段

先只理解概念，不急着扩很多。

### 第二阶段

等 CLI 和工具稳定后，再尝试：

- 浏览 skill
- 安装一个 skill
- 理解 MCP server 配置格式

## Skill 的典型使用方式

官方文档提到，安装后很多 skill 会变成 slash 命令入口。

这意味着它更像：

- 一个可调用模板
- 一个按任务动态加载的操作说明

## MCP 的典型配置理解

一般你会在配置里描述：

- 服务名
- 启动命令
- 参数
- 环境变量

这就意味着你后续排错时要看：

- 命令能不能启动
- 依赖是不是齐
- token 是否有效

## 什么时候再深入

如果你现在还没把基础安装和 CLI 跑顺，先别一头扎进 MCP。它属于“能力扩展层”，不是“点火启动层”。

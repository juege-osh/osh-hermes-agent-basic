# Hermes Gateway 与消息平台接入

## Gateway 是干什么的

Gateway 是 Hermes 和外部消息平台之间的桥。

它的作用是把：

- Telegram
- Discord
- Slack
- 其他消息入口

接到同一个 Hermes 代理核心上。

## 为什么这层很重要

因为很多人最终想要的不是“坐在命令行里和 agent 聊天”，而是：

- 在手机上聊
- 在团队群里聊
- 让机器人长期在线

## 但它为什么也更容易出错

因为这里面会多出很多变量：

- 平台 token
- webhook 或连接方式
- 用户授权
- 会话路由
- 消息格式差异
- 平台限流

## 新手建议顺序

1. 先本地 CLI 跑通
2. 再接一个消息平台
3. 再考虑多平台共存

## 官方常见命令

```bash
hermes gateway
hermes gateway setup
hermes gateway status
```

## Telegram 是很多人的第一站

因为：

- 个人试用方便
- 手机上体验直观
- Bot 生态成熟

但不管接哪个平台，思路都一样：

1. 拿 token
2. 配置 Hermes
3. 启动 gateway
4. 验证消息能来回

## 这里最重要的安全问题

不要把 gateway 当成一个完全裸露、谁都能用的公共入口。

你要考虑：

- 谁能和机器人对话
- 对话能触发哪些工具
- 是否允许执行高风险动作

## 常见失败表现

- 机器人显示在线但没回消息
- 能回消息但内容异常
- 某些用户能用，某些用户不能用
- CLI 正常，平台端异常

## 排错优先级

1. token 是否正确
2. gateway 是否真的启动
3. Hermes 基础模型是否正常
4. 平台授权与 allowlist 是否正确
5. 平台侧限制是否触发

## 什么时候适合上生产环境

至少满足这些再说：

- CLI 稳定
- provider 稳定
- 工具权限有边界
- 会话持久化清楚
- 你知道怎么重启和恢复 gateway

## 这一层的核心认识

Gateway 不是增强“智能”的层，而是增强“接入方式”的层。它本身不替你解决模型、工具和安全边界的问题。
